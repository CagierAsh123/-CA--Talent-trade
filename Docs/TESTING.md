# Talent Trade 回归测试手册

本文档用于验证本轮“交易对象扩展”改动是否稳定：

- 市场 / 直接交易支持：殖民者、殖民地动物、玩家机械体、囚犯
- 租借仍然只支持：普通殖民者
- 不接入 Character Editor，继续使用原有 `PawnSerializer` / `PawnDeserializer` / `TalentTradeManager` 传输链

相关已知缺口见：[RENTAL_GAPS.md](RENTAL_GAPS.md)

---

## 1. 测试前准备

### 1.1 环境

建议至少准备两个 RimWorld 客户端 / 两个玩家：

- A：卖方 / 发起方
- B：买方 / 接收方

两边都需要：

- 安装 Talent Trade 当前编译产物
- 能正常登录 Phinix 聊天
- 能进入同一联机沟通环境

### 1.2 存档准备

建议在 A 侧准备以下 4 类测试对象：

1. 普通殖民者
2. 殖民地动物
3. 玩家机械体
4. 囚犯

并尽量保证每类对象都具备清晰可识别特征，方便验证：

- 名称明显
- 年龄明显
- 身上有少量技能 / traits / 健康状态差异
- 机械体最好已有 mechanitor 归属
- 囚犯最好已处于稳定 prisoner 状态

### 1.3 建议开启的观察项

建议测试时同时观察：

- 游戏内消息
- 开发者日志 / Player.log
- 接收后落地状态
- 保存 / 读档后的状态是否漂移

重点关注是否出现：

- null reference
- mechanitor / overseer 相关报错
- prisoner 丢失状态
- pawn 被错误殖民者化
- 交易成功后发送方自己的 pawn 被错误恢复回来

## 2. 冒烟验证

先做一次最基础检查：

1. 进入游戏。
2. 确认 Talent Trade 面板能正常打开。
3. 确认市场、直接交易、租借三个页签都可进入。
4. 确认市场 / 直接交易的选择文案已变为“可交易单位”语义。
5. 确认租借仍然是“可出租殖民者”语义。

通过标准：

- UI 不缺 key
- 不报红字
- 文案没有明显把动物 / 机械体 / 囚犯继续写成“殖民者专用”

---

## 3. 市场测试

### 3.1 普通殖民者

1. A 打开市场，上架一个普通殖民者。
2. B 刷新市场，确认能看到该条目。
3. B 购买该条目。
4. 观察 B 侧是否通过空投收到 pawn。
5. 保存并读档一次。

验证点：

- A 侧上架后本地 pawn 消失
- B 侧收到后仍是普通殖民者
- faction 正确
- 无异常 guest / prisoner 状态
- 存读档后状态稳定

### 3.2 殖民地动物

1. A 上架一个殖民地动物。
2. B 刷新并购买。
3. 观察动物是否正常空投落地。
4. 保存并读档一次。

验证点：

- 接收后仍是动物，不会变成人类殖民者
- faction 为玩家
- training tracker 正常
- 不带错误 ownership 绑定
- 存读档后无异常

### 3.3 玩家机械体

1. A 上架一个玩家机械体。
2. B 刷新并购买。
3. 观察机械体落地。
4. 保存并读档一次。

验证点：

- 接收后仍是 mech，不会被当成人类 pawn
- 不出现 mechanitor / overseer / relation 相关明显红字
- faction 正确
- 存读档后不报错

### 3.4 囚犯

1. A 上架一个囚犯。
2. B 刷新并购买。
3. 观察囚犯落地。
4. 保存并读档一次。

验证点：

- 接收后仍是 prisoner
- 不会直接并入普通殖民者
- guest/prisoner 状态正确
- faction 为玩家阵营下的囚犯语义
- 存读档后 prisoner 状态不漂移

---

## 4. 直接交易测试

### 4.1 交易窗口基础行为

1. A 向 B 发起直接交易。
2. B 接受。
3. 双方打开交易窗口。

验证点：

- 双方都能看到交易窗口
- “+ 添加单位”按钮正常
- 双方 offer 能正常发送与刷新

### 4.2 四类对象逐一测试

对以下对象分别独立测一次：

- 普通殖民者
- 殖民地动物
- 玩家机械体
- 囚犯

每次流程：

1. A 把一个对象加入报价。
2. B 确认对方报价中能看到正确摘要。
3. 双方锁定交易。
4. 观察双方落地结果。
5. 保存并读档一次。

验证点：

- 加入报价时不会错误拦截支持对象
- 接收方收到后类型保持不变
- 发送方自己的 pawn 不会在“交易成功”后又被错误 restore 回来
- 接收方空投落地正常
- 存读档后状态稳定

### 4.3 双向交换测试

再做一轮双向交换：

- A 给 B：动物 / 囚犯 / 机械体之一
- B 给 A：普通殖民者

验证点：

- 双方都能收到对方 pawn
- 不出现重复生成
- 不出现本地自己发送出去的 pawn 被恢复回来

---

## 5. 租借回归测试（仅普通殖民者）

> 注意：动物、机械体、囚犯不应该进入租借流程。

### 5.1 入口限制

1. 打开租借面板。
2. 点击选择可出租殖民者。
3. 检查可选列表。

验证点：

- 只能看到普通殖民者
- 看不到动物
- 看不到机械体
- 看不到囚犯

### 5.2 浏览列表过滤

1. 让 A 上架一个租借殖民者。
2. B 刷新租借列表。

验证点：

- B 只能看到普通殖民者租借项
- 不存在动物 / 机械体 / 囚犯租借条目

### 5.3 手动租借与归还

1. A 上架普通殖民者。
2. B 租借。
3. B 确认收到 pawn。
4. B 点击归还。
5. A 观察是否收到返还 pawn。

验证点：

- A 能收到返还
- B 侧无明显红字
- A 侧恢复的 pawn 能正常落地

额外观察：

- 当前实现存在已知缺口，需特别检查 B 侧地图上是否还残留租来的 pawn 副本
- 若存在残留，请记录为已知问题，不视为本轮“动物 / 机械体 / 囚犯支持”新增回归

### 5.4 到期 / 死亡 / 复活链路

当前建议把这一组测试视为“审计测试”而不是“本轮扩展功能验收测试”。

执行方法：

1. 完成一次普通殖民者租借。
2. 尝试等待到期，观察是否自动返还。
3. 尝试让租借 pawn 死亡，观察是否触发 owner 侧 revive。

当前预期：

- 这部分链路很可能不完整，详见 [RENTAL_GAPS.md](RENTAL_GAPS.md)
- 若未触发，不代表本轮市场 / 直交易扩展失败，而是租借生命周期本身仍需补全

## 6. 缺少种族 Mod 拦截测试

此项用于验证 race def 仍然是硬门槛。

### 6.1 市场购买拦截

1. A 准备一个依赖额外 race mod 的 pawn。
2. B 不安装该 race mod。
3. A 上架，B 尝试购买。

验证点：

- B 侧购买前被拦截
- 弹出“缺少种族 mod”提示
- 不会生成坏 pawn

### 6.2 直接交易拦截

1. A 向 B 发起交易。
2. A 尝试把缺 race 的 pawn 加入报价。

验证点：

- race check 在加入报价前拦截
- 不会把不兼容 pawn 真正塞进报价

---

## 7. 存档兼容测试

建议至少覆盖以下情况：

1. 市场收到动物后保存 / 读档
2. 市场收到机械体后保存 / 读档
3. 市场收到囚犯后保存 / 读档
4. 直接交易收到上述三类对象后保存 / 读档

验证点：

- 类型不漂移
- prisoner 仍是 prisoner
- mech 不报控制链错误
- animal 不报 ownership / training 异常

---

## 8. 日志检查清单

每轮测试结束后，建议检查日志中是否出现：

- `XmlToPawn failed`
- `pawn.def is null`
- `SpawnViaDropPod failed`
- mechanitor / overseer / relation 相关异常
- prisoner / guest tracker 相关异常

如果只是 `hediff/need warnings` 且 pawn 最终正常落地，可结合当前 `PawnDeserializer` 的说明判断是否为可接受噪音。

---

## 9. 建议验收结论

### 通过标准

若以下都满足，可认为本轮目标基本达成：

- 市场支持：殖民者 / 动物 / 机械体 / 囚犯
- 直交易支持：殖民者 / 动物 / 机械体 / 囚犯
- 租借仅普通殖民者可进入
- 囚犯接收后仍是 prisoner
- 机械体接收后不出现明显控制链错误
- 动物接收后仍是动物
- 交易成功后不会把发送方自己的 pawn restore 回来
- 关键场景保存 / 读档一次后仍稳定

### 暂不作为本轮阻塞项

以下项目建议单独跟进，不阻塞本轮“扩展交易对象”验收：

- 租借到期自动返还
- 租借死亡上报 / revive 回传
- 租借返还时 renter 侧本地副本清理
